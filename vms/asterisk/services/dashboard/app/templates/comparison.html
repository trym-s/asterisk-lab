Latency decision

p50 always, p95 gated by sample floor

How fast each pipeline stage responds, per lane: STT (speech-to-text), LLM (the model composing an answer), Tool (a knowledge-base lookup), TTS (text-to-speech). p50 is the typical case; p95 (the slow-case tail) only renders once there are enough samples (N) to trust it, otherwise it shows "below floor" rather than a misleading number from too few calls. "Source: measured" means a real instrumented duration from the lane itself; "approx" means it was estimated from event timestamps and can be noisy - weight measured numbers more heavily when comparing.

Reliability

comparable outcomes vs lane-specific diagnostics

Did the calls actually work? The left table counts, out of every call in this run, how many: completed without staying stuck, reached all the expected conversation turns (didn't get cut off early or die mid-call), had no error events, had a real (non-empty) transcript on every turn, and didn't double-count a turn. These checks apply identically to both lanes, so this table is the one safe to use when deciding which lane is more reliable. The right table is diagnostics that only make sense for one lane's own architecture (e.g. echo filtering) - informative, but deliberately left out of any head-to-head score since the other lane can't produce a comparable number for it.
